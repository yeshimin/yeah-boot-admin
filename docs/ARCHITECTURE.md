# 架构设计 v1

## 文档目的

本文档用于定义 `yeah-boot-admin` 的第一版架构基线，目标是为后续开发提供稳定约束，避免在项目早期因目录、状态、路由、权限、请求层设计不清而产生持续返工。

本文档优先解决以下问题：

- 代码应该如何分层
- 页面和模块应该如何组织
- 路由、菜单、权限如何统一建模
- 请求层和状态管理各自负责什么
- 哪些约束需要从现在开始执行

## 适用范围

当前文档仅针对前端后台项目架构，不覆盖后端微服务拆分、数据库设计和部署拓扑。

## 当前假设

在用户尚未进一步明确前，本项目 `v1` 默认按以下目标人群设计：

- 主要服务个人开发者和中小团队
- 面向通用后台管理系统，而非重型低代码平台
- 优先保证维护成本、可读性和二次开发效率

如果后续目标改为大型企业级平台，应重新评估插件化能力、多租户边界、微前端拆分和更重的权限模型。

## 当前范围澄清

根据最新确认的信息，当前前端项目不是从零定义一套全新后端契约，而是优先对接一个已经成熟、已在实际工作中使用的开源后端服务项目。

这意味着 `v1` 的目标应调整为：

- 优先适配现有认证、权限、菜单、用户等核心接口
- 以前后端契约对齐为第一优先级
- 在保证高可维护性的前提下，避免为了抽象而脱离现有后端实际返回结构

因此，后续前端架构落地应遵循一个额外原则：

- 先梳理现有后端 API，再决定前端请求层、权限模型和模块拆分的最终细节

## 架构目标

- 单一职责明确，避免页面文件承担请求、权限、表单、状态和展示的全部职责
- 路由、菜单、权限使用同一套元数据模型，避免多处重复维护
- 页面开发默认遵循统一模板，降低新增模块时的心智负担
- 基础设施层集中复杂度，业务页面保持薄而清晰
- 所有高频后台能力都应具备统一入口和统一规范

## 非目标

- 当前阶段不追求一开始就做成插件市场
- 当前阶段不引入微前端
- 当前阶段不追求过度抽象的 DSL 驱动后台
- 当前阶段不为极少数场景设计复杂框架能力

## 总体分层

推荐采用“应用层 + 核心层 + 共享层 + 业务模块层 + 页面层”的结构。

### 1. 应用层 `app`

负责应用启动与全局装配，不承载具体业务。

职责包括：

- 应用入口
- 路由实例创建
- 全局样式注册
- Pinia 注册
- 全局指令、全局组件、全局错误处理挂载

### 2. 核心层 `core`

负责与业务无关但与系统能力强相关的基础设施。

职责包括：

- HTTP 客户端
- 鉴权 token 管理
- 权限判断
- 错误模型
- 应用级常量
- 系统配置模型

核心层的目标是“集中复杂度”，让页面和模块尽量不直接处理底层细节。

### 3. 共享层 `shared`

负责跨业务域复用的通用能力。

职责包括：

- 通用 UI 组件
- 通用 composable
- 通用工具函数
- 通用类型定义
- 指令、格式化、校验器

共享层只放“真正通用”的内容，不接受为了偷懒而塞进去的业务代码。

### 4. 业务模块层 `modules`

负责按业务域组织代码，是后续代码增长的主要承载层。

每个模块内部自行维护：

- API 请求
- 类型定义
- 领域模型转换
- 模块私有组件
- 模块私有 composable
- 模块页面所需的配置常量

例如：

- `modules/system/user`
- `modules/system/role`
- `modules/system/resource`

### 5. 页面层 `pages`

页面层只负责路由入口和页面级编排，不负责复杂业务细节。

页面层应该做的事：

- 读取路由参数
- 组织页面布局
- 调用模块 composable
- 串联搜索区、表格区、弹窗区

页面层不应该做的事：

- 直接写 axios 请求
- 内联大量转换逻辑
- 复制一套又一套相似的增删改查代码

## 路由、菜单、权限统一模型

当前项目已经暴露出菜单、路由、页面入口分别维护的问题。后续必须改为“单一数据源”。

### 原则

- 菜单来源于路由定义，而不是单独维护一份侧边栏配置
- 权限控制来源于路由或资源元数据，而不是散落在页面中的字符串判断
- 页面标题、图标、是否隐藏、是否缓存等信息由路由元数据统一管理

### 建议的路由元数据

推荐统一以下字段：

```ts
type AppRouteMeta = {
  title: string
  icon?: string
  requiresAuth?: boolean
  permission?: string
  hidden?: boolean
  keepAlive?: boolean
  affix?: boolean
  activeMenu?: string
}
```

### 路由分类

- 静态公开路由：登录、404、500
- 静态受保护路由：基础布局、个人中心、基础设置页
- 动态业务路由：根据用户权限加载的模块路由

### 设计约束

- 菜单渲染组件不得自行维护完整菜单树
- 页面标题不得在页面内手动写死
- 面包屑不得依赖额外配置文件重复维护
- 动态路由注入只在权限初始化阶段做一次

## 权限体系设计

权限体系建议拆成三层：

### 1. 登录态

决定“是否可进入受保护区域”。

### 2. 路由权限

决定“用户是否可访问某页面”。

### 3. 资源权限

决定“用户是否可见某按钮、操作、菜单项”。

### 权限数据建议

后端返回的数据建议至少包含：

- 用户基本信息
- 角色集合
- 权限标识集合
- 可访问菜单或可访问路由资源集合

### 前端约束

- 页面按钮权限判断必须通过统一工具或统一指令完成
- 不允许在模板中大量手写重复权限判断表达式
- 不允许在多个页面中复制菜单权限转换逻辑

## 请求层设计

### 基本原则

- 页面和组件不得直接使用裸 `axios`
- 所有请求必须通过统一 HTTP 客户端发出
- 所有模块 API 必须按业务域组织

### 推荐结构

```text
core/http/
  client.ts
  interceptors.ts
  types.ts

modules/system/user/api/
  user.api.ts
```

### 请求层职责

- 统一 base client
- 统一请求头注入
- 统一 token 处理
- 统一错误处理
- 统一响应解包
- 统一超时、取消请求和重试策略边界

### 页面禁止事项

- 页面里直接拼接接口 URL
- 页面里直接处理通用错误提示
- 页面里直接解析后端响应包壳

## 数据模型与类型策略

类型必须靠近业务域，而不是全部堆在全局 `types` 目录。

### 建议做法

- 模块内部类型优先放在模块目录
- 全局共享类型才进入 `shared/types`
- API DTO、页面表单模型、表格展示模型可以分开定义

### 类型分层示意

- `UserDTO`：后端返回结构
- `UserQuery`：查询参数结构
- `UserFormModel`：前端表单结构
- `UserTableItem`：列表展示结构

这样做的目的，是防止一个类型对象承担所有场景，最终谁也不适配。

## 状态管理策略

Pinia 只管理“跨页面共享、具有持久价值、需要统一来源”的状态。

### 应进入全局 Store 的状态

- 登录态与用户信息
- 权限信息
- 布局状态
- 全局字典缓存
- 全局主题配置

### 不应进入全局 Store 的状态

- 单个页面的查询表单
- 单个弹窗的显示开关
- 临时表格选中项
- 仅在当前页面存在的 loading 状态

### 推荐 Store 划分

- `authStore`
- `appStore`
- `permissionStore`
- `dictStore`

### 推荐页面模式

页面复杂逻辑优先抽成模块 composable，例如：

- `useUserTable`
- `useUserFormDialog`
- `useRolePermissionTree`

## 组件体系设计

组件必须分层管理，避免通用组件和业务组件混杂。

### 组件分层

- `shared/ui`：最基础的 UI 封装，如按钮包装、表格包装、空态、状态标签
- `shared/components`：通用但稍有业务语义的复合组件，如分页搜索条、标准工具栏
- `modules/*/components`：某个模块私有组件
- `layouts/*`：布局组件

### 组件约束

- 通用组件不得依赖具体业务模块
- 模块私有组件不要提前抽到全局
- 单文件组件超过合理复杂度时，应拆分 composable 和子组件

## 页面开发标准模板

标准后台列表页建议统一为以下结构：

1. `PageHeader`
2. `SearchForm`
3. `Toolbar`
4. `DataTable`
5. `Pagination`
6. `CreateOrEditDialog`

标准页面应具备：

- 查询参数类型
- 列表数据类型
- API 封装
- 页面 composable
- 权限控制点

## 布局体系设计

后台布局应统一由单一布局状态源驱动。

### 需要统一管理的布局状态

- 侧边栏展开 / 折叠
- 当前激活菜单
- 面包屑
- 标签页导航
- 主题模式
- 设备尺寸状态

### 当前明确决策

- 布局折叠状态后续必须收敛到统一 store
- 头部和侧边栏不能各自维护一份折叠状态

## 错误处理与反馈策略

系统中的错误应分层处理：

- 请求错误：由 HTTP 层统一处理基础逻辑
- 业务错误：由模块层根据业务语义处理
- 页面错误：只处理与当前页面交互强相关的提示

### 约束

- 不允许每个页面重复写一套通用报错逻辑
- 不允许在页面中随意决定登录失效跳转逻辑
- 统一处理 401、403、404、500 等典型状态

## 测试与质量门禁

要实现“高规格”，必须建立质量门禁，而不是只靠人工自觉。

### 最低要求

- `lint`
- `type-check`
- 单元测试
- CI 自动执行校验

### 推荐技术方向

- 单元测试：`Vitest`
- 组件测试：`Vue Test Utils`
- E2E：后续可引入 `Playwright`

### 测试优先级

- 核心 composable
- 权限判断逻辑
- 路由过滤逻辑
- 请求层转换逻辑
- 高复用通用组件

## 文档体系要求

架构文档不是一次性文件，后续调整时必须同步更新。

### 后续必须补齐的文档

- `docs/CODE_STYLE.md`
- `docs/CONTRIBUTING.md`
- `docs/CHANGELOG.md`

### 记录要求

- 重要架构决策必须记录到项目日志
- 架构文档调整后，应同步补一条开发日志

## 从当前原型到目标架构的迁移路径

### 阶段 1：文档和规则落地

- 完成架构文档
- 完成目录规范
- 完成代码规范

### 阶段 2：基础设施收敛

- 重构路由元数据
- 重构布局状态管理
- 引入统一请求层
- 建立权限 store

### 阶段 3：标准样板模块落地

- 选择“用户管理”作为样板模块
- 完成模块拆分、API 分层、页面标准化
- 提炼出可复用的搜索区、表格区、弹窗模式

### 阶段 4：逐步迁移现有页面

- 角色管理
- 资源管理
- 组织架构
- 岗位管理

## 当前决策摘要

- 采用分层 + 按业务域组织模块的架构
- 菜单、路由、权限统一建模，不再多处重复维护
- Pinia 只承载全局共享状态
- 模块页面默认通过 composable 编排，而不是在 SFC 中堆全部逻辑
- 基础设施优先于页面数量
