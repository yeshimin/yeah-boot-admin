# 项目日志

## 基本信息

- 项目名称：`yeah-boot-admin`
- 记录日期：2026-04-03
- 当前阶段：前端后台原型阶段，准备向高规格开源脚手架演进

## 项目背景

当前项目最初参考了若依体系的后台前端思路，但在实际使用过程中，现有框架的开发体验、可维护性和长期演进能力不符合预期。因此，本项目将转向自研，并以开源项目标准重新设计和建设。

本项目的目标不是做一个能跑的后台页面集合，而是沉淀一套可长期维护、可扩展、可复用、可开源协作的前后端脚手架体系。

## 核心目标

### 产品目标

- 打造一套自研的后台管理脚手架，替代原先基于若依的前端方案
- 项目应具备开源发布能力，而不是仅服务于单个私有业务
- 在功能设计上兼顾通用后台场景与二次开发效率

### 工程目标

- 高可维护性：模块边界清晰，目录职责明确，避免页面文件无限膨胀
- 高一致性：代码风格、命名、错误处理、状态管理、接口调用方式统一
- 高可测试性：核心逻辑可单测，关键流程可集成验证
- 高可扩展性：权限、菜单、布局、表格、表单、字典、请求层等均可演进
- 高可读性：新加入的开发者能在较短时间内理解项目结构和开发约定
- 高稳定性：基础设施改动要小心，避免“功能越来越多，项目越来越乱”

### 文档目标

- 文档必须完善，不能等项目做完再补
- 文档应覆盖：项目定位、架构设计、目录规范、开发规范、提交流程、发布流程、常见问题
- 关键设计决策应留痕，避免后续会话或后续开发时反复推翻
- 项目内必须保留持续更新的开发日志，确保跨会话、跨阶段可追溯

## 项目原则

- 先定规则，再写功能
- 先做基础设施，再铺业务页面
- 优先抽象稳定模式，不急于堆页面数量
- 能复用的不重复写，能约束的不用口头约定
- 复杂度要集中在基础层，不要散落在每个页面中
- 所有重要改动都应配套文档和日志记录

## 建议的工程建设方向

### 一、先收敛架构，再继续铺页面

当前仓库已经有后台页面原型，但整体仍偏静态演示。后续不建议继续直接堆业务页，建议先完成以下基础层设计：

- 路由体系：静态路由、权限路由、菜单映射、面包屑、页面标题的统一方案
- 布局体系：头部、侧边栏、标签页、内容区、主题与响应式行为的统一状态管理
- 请求体系：统一的 HTTP 客户端、错误处理、鉴权、重试、取消请求、接口类型定义
- 状态体系：明确哪些进入 Pinia，哪些留在页面内，哪些抽成 composable
- 组件体系：表格、搜索栏、弹窗表单、状态切换、字典显示等后台高频组件标准化
- 权限体系：菜单权限、按钮权限、路由权限、资源标识统一建模

### 二、建立“规范优先”的仓库约束

项目若要长期维护，建议尽快补齐以下约束：

- 目录规范文档
- 命名规范文档
- 组件设计规范
- API 封装规范
- 提交规范
- 代码评审检查清单
- 发布前检查清单

### 三、文档分层建设

建议后续逐步形成以下文档结构：

- `README.md`：项目介绍、快速开始、文档入口

## 质量标准

后续功能建设默认按以下标准推进：

- 页面逻辑尽量拆分，避免单文件组件承载过多职责
- 类型定义要完整，不依赖大面积 `any`
- 重复 UI 交互要抽成可复用组件或 composable
- 所有基础能力要有明确边界和统一入口
- 核心基础模块变更时，同步更新文档
- 每次阶段性改造都要记录到项目日志
